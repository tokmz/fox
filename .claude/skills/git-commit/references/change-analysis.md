# 变更分析协议

## 目标

精准识别当前工作区的所有变更，分类并评估影响范围，为后续提交消息生成提供数据基础。

## 执行步骤

### 1. 收集基础信息

并行执行以下命令：

```bash
# 工作区状态
git status

# 已暂存的变更（将要提交的内容）
git diff --staged

# 未暂存的变更（不会提交但需关注）
git diff

# 近期提交历史（用于风格参考）
git log --oneline -10

# 当前分支
git branch --show-current
```

### 2. 变更分类

根据 diff 内容将变更归类：

| 类型 | type | 判断依据 |
|------|------|---------|
| 新功能 | `feat` | 新增函数、结构体、接口、路由、handler |
| 修复 | `fix` | 修改错误处理、修正逻辑、修复已知 bug |
| 重构 | `refactor` | 改名、移动文件、调整结构、不改变行为 |
| 性能 | `perf` | 算法优化、缓存、减少分配、批量处理 |
| 文档 | `docs` | README、注释、API 文档变更 |
| 样式 | `style` | 格式化、空格、命名风格、不影响逻辑 |
| 测试 | `test` | 新增或修改测试用例 |
| 构建 | `build` | Makefile、Dockerfile、CI 配置、go.mod |
| 回滚 | `revert` | `git revert` 产生的提交 |
| 杂项 | `chore` | 不属于以上任何类型 |

### 3. 影响范围评估

分析变更涉及的模块/层级：

- **文件级**：哪些文件被修改
- **模块级**：涉及哪些 package / module
- **接口级**：是否有 API 接口变更（新增/删除/修改参数）
- **数据级**：是否影响数据库 schema / 数据迁移
- **依赖级**：是否新增/升级了外部依赖

### 4. 敏感信息检查

扫描变更中是否包含：
- 硬编码密钥、token、密码
- `.env` 文件内容
- 数据库连接字符串
- 私钥文件
- 其他 credential

如果发现敏感信息，**立即警告用户并拒绝提交**，不执行后续步骤。

### 5. 拆分建议

如果变更包含多个不相关的关注点（例如：修 bug + 新功能 + 重构），建议用户拆分为多个原子提交。输出拆分方案：

```
建议拆分为 3 个提交：
1. [fix] 修复用户登录 token 过期问题
   文件: auth/token.go, middleware/auth.go
2. [feat] 新增用户头像上传接口
   文件: api/upload.go, service/avatar.go
3. [test] 补充 auth 模块单测
   文件: auth/token_test.go
```

## 输出格式

向用户展示分析结果摘要：

```
📋 变更分析
  分支: feature/user-avatar
  文件: 5 个修改, 3 个新增, 1 个删除
  类型: feat (主导) + fix + test
  范围: auth, api, service
  ⚠️  敏感信息: 未发现
```
